
### 一、 技术栈约定与基础架构

为了保证开发效率和代码的健壮性，建议采用轻量且你非常熟悉的架构：

* **前端视觉交互：** 基于提供的 HTML 原型，补充 Three.js（或类似 WebGL 库）负责 3D 星空的渲染与鼠标射线交互。使用 **Vite + 原生 TypeScript**（不引入 React/Vue，减少构建复杂度），HTML 原型的 CSS 直接复制进项目。
* **后端服务层：** Node.js + Express，负责处理业务逻辑。
* **数据库持久层：** SQLite。单文件结构，免除复杂的环境配置，查询性能应对比赛绰绰有余。
* **部署环境：** 阿里云 ECS 服务器，配合 PM2 进行进程守护，使用 Nginx 做反向代理和 SSL 证书配置，确保前后端通信稳定。

### 二、 团队两人协同分工表

按照功能模块，将两人分为 **"视觉与交互 (A)"** 和 **"数据与服务 (B)"**，两者通过约定好的 API 列表进行对接，互不阻塞。

#### 👩‍💻 开发者 A：前端开发与 3D 交互

**核心职责：** 把 HTML 设计图变成可点、可转的活项目。

* **星空渲染引擎：** 搭建 3D 场景，将星空背景、历史故事星、用户故事星渲染在 3D 球面坐标上。
* **UI 逻辑绑定：** 实现浮层菜单、投递表单、星空标签（如"历史里的星"、"看见大家"）的显示与隐藏逻辑。
* **接口对接：** 使用 `fetch` 或 `axios` 调用后端 API，将拿到的坐标和文本数据映射到 Three.js 的 Mesh 材质上。

#### 👨‍💻 开发者 B：后端逻辑、数据流与运维部署

**核心职责：** 保证数据的真实流转和服务器的稳定运行。

* **数据库设计：** 设计 SQLite 表结构，注意这里**坚决不做任何 public/private 的权限设计**。宇宙星空是平等的，所有挂在星星上的内容默认全量公开展示，这既符合产品"匿名弱连接"的调性，也极大简化了鉴权逻辑。
* **API 接口开发：** 提供星星列表拉取、故事投递、共鸣点亮等 RESTful 接口。
* **真实冷启动数据准备：** 坚决抵制生成毫无意义的 Demo 假数据。编写轻量级爬虫，抓取真实的古诗词、开源社区语录或维基百科的真实星座神话故事，作为初始化注入 SQLite 的"历史星"数据。
* **服务器部署：** 尽早完成 ECS 服务器环境初始化，配置好 Nginx 和绑定的域名，让前端一开始就能调用线上接口进行联调。

##### SQLite 表结构（开发者 B 直接用）

```sql
CREATE TABLE IF NOT EXISTS stars (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  type            TEXT NOT NULL DEFAULT 'user',    -- 'history' | 'user'
  title           TEXT,                            -- 历史星的标题，用户星为 NULL
  content         TEXT NOT NULL,                   -- 故事内容，纯文本，最长 300 字
  resonance_count INTEGER NOT NULL DEFAULT 0,      -- 共鸣次数
  pos_x           REAL NOT NULL,                   -- X 坐标（范围 -300 ~ 300）
  pos_y           REAL NOT NULL,                   -- Y 坐标（范围 -300 ~ 300）
  pos_z           REAL NOT NULL,                   -- Z 坐标（范围 -300 ~ 300）
  created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 加速全量拉取（GET /api/stars 的唯一查询场景）
CREATE INDEX IF NOT EXISTS idx_stars_type ON stars(type);
```

##### 坐标生成算法

新星星的 3D 坐标在 **600x600x600 立方体空间**（pos_x/y/z 各 -300 ~ 300）内均匀随机生成。后端代码：

```js
function generatePosition() {
  const range = 300;
  return {
    x: parseFloat((Math.random() * range * 2 - range).toFixed(1)),
    y: parseFloat((Math.random() * range * 2 - range).toFixed(1)),
    z: parseFloat((Math.random() * range * 2 - range).toFixed(1)),
  };
}
```

> 保留 1 位小数以精简传输体积。不查重（星空里星星重叠是自然的）。

##### 冷启动数据规则

| 字段 | 爬虫抓取时的处理 |
|---|---|
| `type` | 全部为 `"history"` |
| `title` | 取诗句/神话的题目或首句（如"静夜思"、"牛郎织女"） |
| `content` | 原文内容，超过 300 字截断 + "..." |
| `resonance_count` | 随机 5~200（模拟已有共鸣） |
| `position` | 用上面的 `generatePosition()` |

**爬虫数据源建议（至少 20 条）：**
1. 古诗词：从 [今日诗词 API](https://www.jinrishici.com/) 或古诗文网抓取
2. 星座神话：维基百科"星座"条目下的神话故事
3. 开源社区语录：如 v2ex.com 的"自言自语"节点热门内容
4. 如果时间紧，至少保证 10 条诗词 + 10 条星座故事

##### Nginx 最小反向代理配置（跨域解决）

```nginx
server {
    listen 80;
    server_name your-domain.com;

    # 反向代理到 Node.js 后端
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;

        # CORS 头（前端跨域调用必须）
        add_header Access-Control-Allow-Origin * always;
        add_header Access-Control-Allow-Methods 'GET, POST, OPTIONS' always;
        add_header Access-Control-Allow-Headers 'Content-Type' always;

        if ($request_method = 'OPTIONS') {
            return 204;
        }
    }

    # 前端静态资源（构建后由 Vite 生成到 dist/）
    location / {
        root /var/www/star-empathy/dist;
        try_files $uri $uri/ /index.html;
    }
}
```

### 三、 统一 API 接口契约 (API Contract)

这是团队并行开发的核心。开发者 B 优先用最完整的代码把这三个接口跑通并部署，开发者 A 直接对着这个数据结构写渲染逻辑。

#### 通用错误码规范

所有响应统一格式：`{ code, message, data }`

| code | 含义 | 触发场景 |
|---|---|---|
| 200 | 成功 | 正常响应 |
| 400 | 请求参数错误 | `content` 为空、超过 300 字、缺少必填字段 |
| 404 | 资源不存在 | `resonate` 时 `id` 对应星星不存在 |
| 500 | 服务器内部错误 | 数据库异常、未知错误 |

**错误响应示例：**
```json
{
  "code": 400,
  "message": "content 不能为空且不超过 300 字",
  "data": null
}
```

#### 1. 获取全景星空数据 (`GET /api/stars`)

拉取所有的星星数据，一次性交给前端进行 3D 渲染。

* **请求参数：** 无
* **返回格式 (JSON)：**

```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": 1,
      "type": "history",
      "title": "天狼星的守望",
      "content": "古人把神话写进星座...",
      "resonance_count": 128,
      "position": { "x": 120.5, "y": 45.2, "z": -89.0 },
      "created_at": "2023-08-22T14:00:00Z"
    },
    {
      "id": 2,
      "type": "user",
      "title": null,
      "content": "今天晚上又加班了，但回家的路上看到了月亮。",
      "resonance_count": 12,
      "position": { "x": -50.1, "y": 210.0, "z": 15.6 },
      "created_at": "2026-07-13T10:30:00Z"
    }
  ]
}
```

#### 2. 投递心事/挂上故事 (`POST /api/stars/story`)

用户匿名写下心事，投递到星空。后端负责接收文本，并计算出一个随机的 3D 坐标存入数据库。

> **v1 版本不做 AI 分析**（情绪标签/回信放到 Phase 2，避免 hackathon 时间不够）。

* **请求参数 (Body):**

```json
{
  "content": "今天参加大赛提交了初步思路，希望能有个好结果。" 
}
```

* **字段校验规则（后端必须做）：**
  - `content`：必填，纯文本，trim 后长度 1~300 字符
  - 拒绝 HTML 标签（strip 或转义 `< > &`）
  - 校验失败返回 `code: 400`

* **返回格式 (JSON):** 返回新生成的星星实体，前端拿到后动态在画面中追加一颗星。

```json
{
  "code": 200,
  "message": "故事已化作星光",
  "data": {
    "id": 105,
    "type": "user",
    "content": "今天参加大赛提交了初步思路，希望能有个好结果。",
    "resonance_count": 0,
    "position": { "x": 15.2, "y": -88.1, "z": 200.5 },
    "created_at": "2026-07-13T11:13:37Z"
  }
}
```

#### 3. 点亮共鸣 (`POST /api/stars/:id/resonate`)

用户遇到相似故事时点击"我也感同身受"。极其轻量的接口，后端让对应 ID 的 `resonance_count` +1。

* **请求参数：** URL 传参 `id`
* **返回格式 (JSON):**

```json
{
  "code": 200,
  "message": "共鸣已点亮",
  "data": {
    "id": 105,
    "resonance_count": 1
  }
}
```

### 四、 执行 Timeline 建议

为了保证代码最完整且不易出错，建议分三步走：

1. **Day 1：基建与跑通链路**
* B：建 SQLite 库，写好三个 API 的核心逻辑代码（不精简，写全错误处理），直接扔到 ECS 上跑起来，用 Nginx 暴露出去。写爬虫抓取 20 条真实的古文或诗句注入数据库。
* A：搭前端工程（`npm create vite@latest star-empathy -- --template vanilla-ts`），引入提供的 HTML/CSS，引入 Three.js 画一个简单的球体背景。


2. **Day 2-3：视觉与逻辑拼装**
* A：对接 B 的线上真实接口。通过坐标系数学公式，把获取到的数据精确映射成点阵粒子或 3D 球体；写好"点击星星 -> 弹出故事"、"点击提交 -> 发送 API" 的前端逻辑。


3. **Day 4：联调与体验打磨**
* 前后端一起，重点打磨 3D 漫游的拖拽手感、点击命中率（Raycaster 射线检测优化），以及 Nginx 反向代理配置下的跨域和 HTTPS 稳定性测试。
